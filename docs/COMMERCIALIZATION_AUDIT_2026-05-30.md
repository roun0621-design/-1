## Executive Summary (TL;DR)

PACE RISE는 **육상경기 운영 OS**로서 기능 깊이는 이미 시장의 어떤 경쟁 제품보다 깊습니다 (NR/DR/CR 자동감지, 페이싱 라이트, 콜룸 모니터, 통백업/복원, 외부 API, PWA 오프라인 큐 등). 그러나 현재는 **"잘 만들어진 1대회 운영 도구"** 단계이며, **"수십 단체가 동시에 쓰는 SaaS"** 가 되려면 **6개 영역에서 구조적 변환**이 필요합니다.

## 2. 🔴 P0 — 상용화 전 필수 (Critical Blockers)

### 2.1 멀티테넌시 부재 — 가장 큰 구조적 한계

**현재 상태**:
- 모든 데이터가 한 SQLite 파일에 평면적으로 저장
- `organization_id` / `tenant_id` 컬럼 없음
- "한국실업육상연맹"과 "광주시육상연맹"이 같은 시스템 쓰면 데이터 섞임

**문제 시나리오**:
- A연맹이 결제하고 가입 → B연맹도 가입 → 서로의 대회 데이터가 보임
- 한 단체의 운영키가 다른 단체 데이터를 만진다
- B2B 영업이 사실상 불가능

**필요 작업**:
```
1. organization (조직) 테이블 신설
   - id, name, slug, plan_tier, billing_status, owner_user_id,
     created_at, settings_json, brand_logo_url

2. user (사용자) 테이블 신설
   - id, email, password_hash, name, role,
     organization_id (FK), last_login_at, mfa_enabled

3. 모든 주요 테이블에 organization_id 추가
   - competition, athlete, event, certificate_template,
     federation_list, home_popup, system_config 등

4. Row-Level Security 미들웨어
   - req.user.organization_id 기반 자동 필터
   - SQL에 WHERE organization_id = ? 자동 주입

5. 도메인/서브도메인 라우팅 (선택)
   - ktfl.pacerise.app, gwangju.pacerise.app
   - 또는 paths: /org/ktfl/...
```

**임팩트**: 멀티테넌시 없이는 **SaaS가 아니라 단일 설치형 도구**일 뿐입니다.

### 2.2 인증·세션·권한 시스템 재설계

**현재 상태**:
- 운영키/관리자키/기록위원키만 존재 (3종의 비밀번호)
- 세션/JWT 없음 → URL 쿼리에 `?key=1234` 평문 전송이 흔함
- localStorage에 키 저장 → XSS 1회로 즉시 탈취
- 사용자 식별 불가 (누가 어떤 작업을 했는지 추적 한계)

**보안 위험 분석**:

| 위험 | 현 상태 | 권장 상태 |
|---|---|---|
| 비밀번호 평문 URL 전송 | 🔴 일부 라우트 잔존 | POST body / Header only |
| 세션 만료 | ❌ 없음 (영구) | JWT 8시간 + Refresh |
| MFA (2단계 인증) | ❌ 없음 | TOTP / SMS OTP |
| 비밀번호 정책 | ❌ 4자 이상만 | 강도 검증 + 주기적 변경 |
| Brute-force 방어 | 🟡 일부 rate-limit | 계정 잠금 + IP 차단 |
| 감사 로그 (누가 무엇을) | 🟡 audit_log 6건만 | 모든 변이 작업 기록 |
| CSRF 보호 | ❌ 없음 | SameSite 쿠키 + 토큰 |
| 비밀번호 해시 | ✅ bcrypt (관리자만) | 모든 키 bcrypt 통일 |

**필요 작업**:
```
1. 이메일/비밀번호 회원가입 + 로그인 (전통적 방식)
2. JWT Access (15분) + Refresh (7일) 토큰
3. HttpOnly + Secure + SameSite=Strict 쿠키
4. 역할 기반 접근 제어 (RBAC):
   - org_owner / org_admin / operator / record_officer /
     judge / public_viewer
5. 권한 매트릭스를 코드가 아니라 DB로 (변경 가능)
6. 감사 로그 의무화 — req → 응답 → 변경 전후 diff 저장
7. 비밀번호 재설정 (이메일 링크), 계정 잠금
8. (선택) Kakao/Naver 소셜 로그인 — 한국 시장 진입 장벽 ↓
```
